## Remediated Finding

### EMB-01 · Major · `@uvrn/core` declared as peer dependency but never imported

**File**: `uvrn-embed/package.json:21`
**Evidence**: `grep -r '@uvrn/core' uvrn-embed/src/` returns zero matches.

`@uvrn/embed` declares `"@uvrn/core": ">=1.0.0"` in `peerDependencies`, but no source file in the package imports anything from `@uvrn/core`. Consumers are forced to install a package they don't use. This is the same class of issue as IDN-01 (Wave 3) — a phantom peer dependency.

The build prompt specifies `Peer: @uvrn/core types only (ESM)`, but the implementation defines `BadgeData`, `BadgeProps`, and `BadgeStatus` locally in `src/types/index.ts` rather than importing or re-exporting from `@uvrn/core`. Either the types should be imported from core (making the peer dep real), or the peer dep should be removed (making the package truly standalone).

**Impact**: Violates the package independence principle. Consumers get an unnecessary install requirement. Published `.d.ts` files don't reference `@uvrn/core`, so the peer dep adds confusion without type-level benefit.

**Fix**: Remove `"@uvrn/core": ">=1.0.0"` from `peerDependencies` and `"@uvrn/core": "workspace:*"` from `devDependencies`. Update README minimal install to drop `@uvrn/core`. If the intent is to share types with core in the future, document that in CHANGELOG as a planned v1.1 change.

**Remediation applied**:
- Removed `@uvrn/core` from `uvrn-embed/package.json`
- Removed `@uvrn/core` from the `uvrn-embed/README.md` dependency docs
- Added a remediation note to `uvrn-embed/CHANGELOG.md`

**Current status**: Resolved.

## Accepted Non-Blocking Notes

### EMB-02 · Minor · Cache is keyed by `claimId` only — different `apiUrl` values share the same entry

**File**: `uvrn-embed/src/runtime/badge.ts:27–29`
**Lines**: `cacheKey()` returns `claimId` directly.

If two badges on the same page render the same `claimId` against different `apiUrl` endpoints (e.g. production vs. staging), they will share a single cache entry. The first fetch wins; the second badge silently shows data from the wrong API.

**Impact**: Incorrect badge data when the same `claimId` is fetched from multiple API endpoints on the same page. Unlikely in typical usage, but architecturally unsound.

**Disposition**: Accepted for v1. The current Wave 4 build prompt explicitly specifies a cache keyed by `claimId`, so this is a design note rather than a contract violation.

### WCH-01 · Minor · Module-level `sequence` counter leaks across `Watcher` instances

**File**: `uvrn-watch/src/watcher/Watcher.ts:21`
**Line**: `let sequence = 0;`

The subscriber ID counter is a module-level `let`, not an instance field. If a consumer creates multiple `Watcher` instances (e.g. one per claim group), subscriber IDs increment globally across all instances. IDs like `sub_0001` through `sub_0010` might be split across two watchers that have no relationship.

**Impact**: No correctness issue — subscriber IDs are only used as internal map keys within a single Watcher instance. But it makes debugging confusing if IDs appear non-sequential within a single watcher, and it's a minor information leak about global usage.

**Disposition**: Accepted as cosmetic. No correctness impact; subscriber IDs remain unique.

### WCH-02 · Minor · `flushPromises()` in tests may be insufficient for `Promise.allSettled` chains

**File**: `uvrn-watch/tests/watch.test.ts:32–35`
**Lines**: `flushPromises` resolves only two microtick levels.

The `handleThreshold` → `dispatchAlert` → `Promise.allSettled(buildTargets(...))` chain is at least three levels deep. `flushPromises` does `await Promise.resolve(); await Promise.resolve();` — two microticks. Currently passes because the mock `fetch` resolves synchronously, but if mock behavior changes (e.g. to use `mockResolvedValueOnce` with async resolution), tests could become flaky.

**Impact**: Test reliability under future mock changes. Not a production issue.

**Disposition**: Accepted as a test-hygiene note only. Not a production issue and not blocking Wave 4.

### META-01 · Minor · `dist/` directories present in untracked packages — will be staged if `git add uvrn-watch/ uvrn-embed/`

**File**: `uvrn-watch/dist/`, `uvrn-embed/dist/`

Both packages have `dist/` directories from the build. The root `.gitignore` excludes `dist/` globally, so a bare `git add .` is safe. However, if someone runs `git add uvrn-watch/` or `git add uvrn-embed/` with `-f`, dist could be committed. Neither package has a local `.gitignore` as a second defense layer.

**Impact**: Low — root gitignore covers it. But other packages in the monorepo (e.g. `uvrn-signal`) have per-package `.gitignore` files as defense in depth.

**Disposition**: Not actionable for this remediation. The root `.gitignore` already excludes `dist/` globally, which satisfies the protocol requirement.
