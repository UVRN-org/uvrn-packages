## Findings

### CONS-01 · Major · `@uvrn/agent` types in public API but not declared as peer dependency

**File**: `uvrn-consensus/src/types/index.ts:1`, `uvrn-consensus/package.json`
**Lines**: types/index.ts L1, aggregation.ts L2

`@uvrn/consensus` re-exports `FarmResult` and `FarmSource` from `@uvrn/agent` via its public types barrel, and `aggregation.ts` imports them at the type level for runtime shape access. However, `@uvrn/agent` appears only in `devDependencies` — not in `peerDependencies`.

**Impact**: When a consumer installs `@uvrn/consensus` outside the monorepo, TypeScript will fail to resolve `FarmResult` and `FarmSource` unless the consumer independently installs `@uvrn/agent`. The published `.d.ts` files will reference `@uvrn/agent` types that don't exist in the consumer's `node_modules`.

**Fix**: Add `"@uvrn/agent": ">=1.0.0"` to `peerDependencies` in `uvrn-consensus/package.json`. Update the README minimal install section to include `@uvrn/agent`.

### CONS-02 · Minor · `stats()` calls `#rankedSources()` redundantly when called after `buildBundle()`

**File**: `uvrn-consensus/src/engine/ConsensusEngine.ts`
**Lines**: L41, L68–69

`buildBundle()` calls `this.#rankedSources()` at L21, and `stats()` calls it again at L41. The extraction, parsing, deduplication, and sorting pipeline runs twice. Not a correctness issue, but wasteful for large source arrays.

**Impact**: Performance only. No behavioral difference.

**Suggestion**: Cache `#rankedSources()` lazily after first call. Low priority — v1 source counts are small.

### IDN-01 · Major · `@uvrn/adapter` peer dependency is unused — phantom re-export

**File**: `uvrn-identity/src/types/index.ts:1`, `uvrn-identity/package.json`
**Lines**: types/index.ts L1

`@uvrn/identity` declares `@uvrn/adapter` as a peer dependency and re-exports `DRVC3Receipt` from it. However, `DRVC3Receipt` is not consumed by any code in the package — not in `IdentityRegistry.ts`, not in `reputation.ts`, not in tests.

**Impact**: Consumers must install `@uvrn/adapter` to use `@uvrn/identity`, even though nothing in the package actually uses it. This violates the package independence principle (CLAUDE.md) and adds a phantom dependency.

**Fix**: Remove the `export type { DRVC3Receipt } from '@uvrn/adapter'` line and remove `@uvrn/adapter` from both `peerDependencies` and `devDependencies`. If a future version needs it, add it back intentionally. Update README minimal install to drop `@uvrn/adapter`.

### IDN-02 · Minor · `determineLevel()` returns `'new'` for receipts >= 10 with score < 60

**File**: `uvrn-identity/src/registry/reputation.ts`
**Lines**: L3–18

The level function's fallthrough path returns `'new'` for signers with `receipts >= 10` but `score < 60`. This is a gap between the build prompt spec (which defines four distinct levels) and the implementation. A signer with 50 receipts and score 45 gets level `'new'`, which is semantically misleading.

**Impact**: Minor behavioral surprise. The spec doesn't define a fifth level, so `'new'` is the only remaining option. But the threshold descriptions in CLAUDE.md say `new: receipts < 10 (any score)` — a signer with 50 receipts labeled `'new'` contradicts that definition.

**Suggestion**: Either add a fifth level (e.g. `'low'`), or document this explicitly as a known gap in the v1 README and CHANGELOG. Not a regression — the spec doesn't account for this quadrant.

### IDN-03 · Minor · Floating-point accumulation in `record()` accuracy/canonRate tracking

**File**: `uvrn-identity/src/registry/IdentityRegistry.ts`
**Lines**: L72–77

`record()` reconstructs running counts by multiplying `existing.accuracy * existing.receipts` and `existing.canonRate * existing.receipts`. Over many records, floating-point error accumulates because the intermediate fractional values lose precision when round-tripped through multiplication.

**Impact**: After hundreds of `record()` calls on the same signer, the accuracy and canonRate values may drift slightly from the true values. Unlikely to affect level thresholds in practice, but makes deterministic replay harder.

**Suggestion**: Store raw counts (`accurateCount`, `canonizedCount`) in `ReputationScore` or `IdentityStore` instead of reconstructing them from fractions. Low priority for v1 since the additive-only model means scores trend upward regardless.

### TML-01 · Minor · `buildChartData()` double-buckets snapshots

**File**: `uvrn-timeline/src/timeline/chart.ts`
**Lines**: L69–76

`buildChartData()` calls `bucketSnapshots()` on its `snapshots` parameter. However, `Timeline.query()` at L134 already calls `bucketSnapshots()` and passes the result as `sampledSnapshots` to `buildChartData()` via the `chart()` closure. The snapshots are bucketed twice.

**Impact**: No correctness issue — bucketing an already-bucketed array is idempotent. Pure wasted computation.

**Fix**: Remove the `bucketSnapshots()` call inside `buildChartData()` and operate directly on the input array, or document that `buildChartData()` expects unbucketed input and remove the pre-bucketing in `Timeline.query()`.

### TML-02 · Minor · `#fetchTimeline()` called twice for snapshots+canonEvents in apiUrl mode

**File**: `uvrn-timeline/src/timeline/Timeline.ts`
**Lines**: L148–159

When using `apiUrl` mode, `#fetchSnapshots()` and `#fetchCanonEvents()` each independently call `#fetchTimeline()`, resulting in two identical HTTP requests to the same endpoint. The response contains both `snapshots` and `canonEvents`, but only one field is extracted per call.

**Impact**: Double network round-trip in apiUrl mode. Not triggered in the primary store-based path.

**Fix**: Call `#fetchTimeline()` once, then extract both fields from the single response. Low priority — apiUrl mode is documented as optional/additive.

### CMP-01 · Minor · `@uvrn/core` and `@uvrn/drift` are peer deps but no types from them are imported

**File**: `uvrn-compare/package.json`, `uvrn-compare/src/types/index.ts`

`@uvrn/compare` declares `@uvrn/core` and `@uvrn/drift` as peer dependencies, but the source code doesn't import anything from either package. The `CompareEngine` operates on `unknown[]` inputs and does its own field extraction via the `parseReceipt()` helper. The `@uvrn/test` factory (`mockDriftSnapshot`) is a devDependency used only in tests.

**Impact**: Consumers must install `@uvrn/core` and `@uvrn/drift` to use `@uvrn/compare`, even though the package works entirely on structurally-typed `unknown` inputs at runtime. This is a mild violation of the package independence principle.

**Counterpoint**: The build prompt explicitly specifies these as peer deps, and they document the intended input shapes. The overhead is low since most consumers already have both. Keeping them communicates intent. This finding is informational — follow the build prompt's decision unless the project owner disagrees.

---

### META-01 · Minor · `workspace:*` devDependencies must be converted before npm publish

**File**: All four `package.json` files
**Lines**: `devDependencies` sections

All four packages use `"workspace:*"` for their `@uvrn/*` devDependencies. pnpm will convert these to real semver ranges at publish time if using `pnpm publish`, but `npm publish` from the tarball will fail.

**Impact**: No runtime effect (devDependencies aren't installed by consumers). But the NPM publish checklist in CLAUDE.md explicitly says "package.json has no `workspace:` deps (use semver)". This should be cleaned before publish.

**Fix**: Replace `workspace:*` with `>=1.0.0` (or the actual version) in devDependencies before publishing. Alternatively, rely on `pnpm publish` which handles conversion automatically.

## Residual Risks

1. **Consensus numeric extraction is fragile.** `extractMetricValue()` takes the first regex number match from `title + snippet`. A source like "Founded in 2008, reserves are 100M" extracts `2008`, not `100`. This is documented behavior and acceptable for v1, but production usage on diverse source text will hit edge cases. A follow-up could use a more context-aware extraction strategy.

2. **Identity accuracy tracking assumes `consensusVScore` is available.** The `ReputationActivity.consensusVScore` field is required (not optional), meaning callers must always supply it. If the consensus score isn't available at the time of recording, the caller must fabricate a value or skip recording. The build prompt approved making it required, but the ROADMAP spec had it as optional. Worth revisiting for v1.1.

3. **Compare divergence detection is O(n²) in number of timestamps.** The `calculateDivergenceAt()` function creates a merged timestamp set from all receipts and sweeps through it. For large receipt histories (thousands of points), this could be slow. Acceptable for v1.

4. **Timeline apiUrl mode lacks caching, retry, and auth.** The `#fetchTimeline()` implementation does a bare `fetch()` with no auth headers, no retry logic, and no response caching. This is consistent with the v1 "reference implementation" philosophy, but should be documented as not production-ready for high-throughput use.
